## Bugs — RFC violations

### Bug 1 — `marshalCertOrEncCert` uses IMPLICIT encoding; RFC requires EXPLICIT

**File:** `backend/pkg/controllers/cmp_asn1.go`, function `marshalCertOrEncCert`
**RFC:** RFC 4210 Appendix F (`DEFINITIONS EXPLICIT TAGS`); RFC 9480 ASN.1 module (`DEFINITIONS EXPLICIT TAGS`)

Both RFC 4210 and its update RFC 9480 declare `DEFINITIONS EXPLICIT TAGS` for the PKIXCMP module. The `CertOrEncCert` CHOICE:

```asn1
CertOrEncCert ::= CHOICE {
    certificate     [0] CMPCertificate,
    encryptedCert   [1] EncryptedValue
}
```

With EXPLICIT TAGS, `certificate [0] CMPCertificate` must encode as an outer `[0]` wrapper containing the inner Certificate SEQUENCE:

```
Correct (EXPLICIT):   A0 <len> { 30 <len> <certificate content> }
Produced (IMPLICIT):  A0 <len> <certificate content>
```

The current code passes `certSeq.Bytes` (SEQUENCE content only) into the `asn1.RawValue`, which produces the IMPLICIT form. It should pass `certSeq.FullBytes` (full TLV including the SEQUENCE tag) to produce the EXPLICIT form.

**Impact:** Strict ASN.1 parsers on EE devices will fail to read the issued certificate from the IP/CP/KUP response.

**Fix:** In `marshalCertOrEncCert`, change `Bytes: certSeq.Bytes` to `Bytes: certSeq.FullBytes`.

### Bug 2 — `decodeCertReqMessages` / `rewrapBodyAsSequence` assumes IMPLICIT PKIBody; RFC requires EXPLICIT

**Files:** `backend/pkg/controllers/cmp.go` (`decodeCertReqMessages`), `cmp_asn1.go` (`rewrapBodyAsSequence`)
**RFC:** RFC 4210 Appendix F (`DEFINITIONS EXPLICIT TAGS`)

Because RFC 4210 uses EXPLICIT TAGS, each PKIBody CHOICE alternative `[n] TYPE` wraps the inner SEQUENCE in an outer `[n]` tag. Therefore `rawMsg.Body.Bytes` already contains the **full** CertReqMessages SEQUENCE TLV (starting with `30 xx …`).

`rewrapBodyAsSequence` unconditionally prepends another SEQUENCE header, producing a double-wrapped structure:

```
Body.Bytes (from wire, EXPLICIT):  30 <len> <CertReqMessages content>
After rewrap:                      30 <newlen> { 30 <len> <content> }  ← wrong
```

`asn1.Unmarshal` then fails because the first element of the outer SEQUENCE is another SEQUENCE rather than the expected INTEGER `certReqId`.

This bug causes `handleEnroll` and `handleReenroll` to reject **all** RFC-compliant CMP messages from real clients (e.g. OpenSSL). The comment in `rawPKIMessage` ("Per RFC 4210 Appendix F (IMPLICIT TAGS module default)") incorrectly describes the module tagging.

Evidence: the test helper `extractSubjectCNFromBody` correctly handles EXPLICIT PKIBody by treating `Body.Bytes` as a full SEQUENCE and peeling it directly, without any rewrapping — and it works on the real sample files.

**Impact:** Production enrollment (`handleEnroll` / `handleReenroll`) silently rejects all requests from standard CMP clients.

**Fix:** Remove `rewrapBodyAsSequence` from `decodeCertReqMessages` and decode `body.Bytes` directly as `[]cmp.CertReqMessage`. Note that gocmp's `CertTemplate` struct also has a secondary IMPLICIT/EXPLICIT mismatch for CHOICE-typed fields (e.g. `subject [5] Name` where `Name` is a CHOICE, which must be EXPLICIT even inside an IMPLICIT TAGS module per X.680 §30.6). A complete fix requires either a custom `CertTemplate` decoder or a fork of gocmp.

### Bug 3 — `certStatusASN1` missing `hashAlg [0]` field

**File:** `backend/pkg/controllers/cmp_asn1.go`, struct `certStatusASN1`
**RFC:** RFC 9480 §2.10

RFC 9480 §2.10 updated `CertStatus` to add an optional `hashAlg` field:

```asn1
CertStatus ::= SEQUENCE {
    certHash    OCTET STRING,
    certReqId   INTEGER,
    statusInfo  PKIStatusInfo OPTIONAL,
    hashAlg [0] AlgorithmIdentifier OPTIONAL   -- added by RFC 9480
}
```

The current struct:

```go
type certStatusASN1 struct {
    CertHash   []byte
    CertReqID  int
    StatusInfo cmp.PKIStatusInfo `asn1:"optional"`
    // hashAlg missing
}
```

RFC 9480 §2.10: *"for some cases, e.g., for EdDSA, the hashAlg MUST be used to specify the hash algorithm to be used for calculating the certHash value."*

**Impact:** CertConf messages from EEs using EdDSA certificates will fail hash verification if the EE sends `hashAlg`. For RSA/ECDSA (current supported key types) the field is never needed.

**Fix:** Add `HashAlg pkix.AlgorithmIdentifier \`asn1:"optional,tag:0"\`` to `certStatusASN1` and use it when present to select the hash algorithm instead of the hardcoded SHA-256.

## Design gaps (known / intentional)

### Proof-of-Possession (POP) not validated

**File:** `backend/pkg/controllers/cmp.go`, `buildSyntheticCSR` and callers
**RFC:** RFC 4211 §4; RFC 9483 §5.2.3

RFC 4211 §4: *"Any CRP referencing this document MUST enforce POP by some means."*
RFC 9483 §5.2.3: the PKI management entity MUST verify POP and respond with `badPOP` on failure.

The current implementation creates a synthetic CSR with a dummy zero-byte signature and relies on `VerifyCSRSignature=false` in the DMS `EnrollmentSettings` to bypass signature verification. The `popo` field from `CertReqMessage` is parsed but never validated.

**Impact:** An attacker on the same network could request a certificate for an arbitrary public key without possessing the corresponding private key.

**Acceptable in current deployment context?** Only if the CMP endpoint is protected by mutual TLS or is unreachable from untrusted hosts, AND the DMS enrollment policy provides equivalent identity assurance (e.g. pre-shared credentials, allowlisted device IDs). This MUST be documented in the deployment guide.

**Fix path:** Parse `popo.POPOSigningKey.signature` from the `CertReqMessage`, verify it against the public key in `CertTemplate.PublicKey`, and return a `badPOP` error response on failure.

## ASN.1 tagging reference

| Module | Tagging default | Key types affected |
|--------|----------------|--------------------|
| RFC 4210 / RFC 9480 (PKIXCMP) | **EXPLICIT TAGS** | PKIBody CHOICE [0..24], CertOrEncCert [0]/[1] |
| RFC 4211 (PKIXCRMF) | **IMPLICIT TAGS** | CertTemplate fields [0..9] |
| X.680 §30.6 | CHOICE always EXPLICIT | `subject [5] Name` (Name is CHOICE) even inside RFC 4211 IMPLICIT module |

This explains why `subject [5]` in the sample CertTemplate messages uses an inner SEQUENCE wrapper (EXPLICIT, required for CHOICE) while `publicKey [6]` does not (IMPLICIT for SEQUENCE type).
