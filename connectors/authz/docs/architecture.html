Go Module

github.com/lamassuiot/authz

Authz Connector — Architecture

Fine-grained, policy-driven authorization engine for the Lamassu IoT platform. Supports OIDC and X.509 principal matching, entity-relationship-based SQL filter generation, capability queries, and both embedded and remote deployment modes.

Service Layer

pkg/service/authz.go
AuthzImplementation (Service)
Implements core.AuthzEngine. Adds the principal-resolution layer on top of the Engine: given a principal ID or auth material, loads the appropriate PolicyRegistry via IdentityResolver, then delegates to Engine. Logs the final authorization decision with structured fields and the request trace-ID.

pkg/service/identity.go
Identity Resolver
Single entry point for "auth material → policies" resolution. Calls MatchService to find matching active principals, then loads all their granted policies into a PolicyRegistry. Returns ErrNoMatch when nothing matches, allowing callers to distinguish 401 from 403. Uses local private interfaces (principalMatcher, policyLoader) to avoid circular imports.

Façade over GormPrincipalStore (pkg/store) and MatchService. Provides all principal CRUD and policy grant/revoke operations. Also exposes NewIdentityResolver() to wire a resolver using this manager's store as the GrantStore.

Engine Layer

pkg/engine/engine.go
Authorization Engine
The core decision maker. Validates namespace/schema/entity, classifies actions as global (no DB query) or atomic (SQL existence check), builds SQL filters via FilterGenerator, and executes against the appropriate PostgreSQL database. Accepts a logrus.Entry via functional option; uses helpers.ConfigureLogger(ctx) to attach the request trace-ID to every log line.
EngineAuthorizeGetListFilterEngineOptionWithLogger
pkg/engine/capabilities.go
Capability Evaluator
Pure engine methods for computing what a principal CAN do given an already-loaded PolicyRegistry. GetGlobalCapabilities scans all schemas for granted global actions without DB queries. GetEntityCapabilities probes each atomic action on a concrete entity. The ForPrincipal variants (which need PrincipalManager/PolicyManager) live in pkg/service/capabilities.go to avoid circular imports.
GlobalCapabilitiesEntityCapabilitiesGetEntityCapabilitiesBatchMergeGlobalCapabilities
pkg/engine/filter.go
SQL Filter Generator
Translates policy rules into SQL JOINs and WHERE conditions. Handles three access patterns: direct ID grants (IN clause), cascading FK relations (JOIN chain computed via AuthorizationGraph path-finding), and attribute-based column filters. Produces both list filters (open-ended) and check filters (entity-key-bounded existence queries).
FilterGeneratorFilterResultGenerateListFilterGenerateCheckFilterbuildPathFilter
pkg/engine/graph.go
Authorization Graph
Builds a directed entity-relationship graph from policy rules and schema relation configs. Edges represent foreign-key relationships between entity types. DFS path-finding discovers cascading access paths (e.g. user → organization → device), which FilterGenerator converts into JOIN chains.
AuthorizationGraphGraphNodeGraphEdgeFindPathsBetween
pkg/engine/schema.go
Schema Registry
Loads entity schemas from JSON files. Each SchemaDefinition describes a database table: its qualified name, primary key columns, foreign-key relations to other entity types, which actions are "atomic" (need DB check) vs "global" (no DB check), and which columns can be used in attribute filters. Supports multi-schema environments.
SchemaRegistrySchemaDefinitionFilterableFieldRelationConfig
pkg/engine/policy.go
Policy Registry & Validation
In-memory registry of all policies active for a request. Validates policy structure (rule fields, relation cycles, column filter types), provides schema-matching helpers (wildcards, namespace checks), and exposes rule iteration for engine and filter generator. A new PolicyRegistry is built per-request from blob storage.
PolicyRegistryAddPolicyGetRulesruleMatchesSchemawildcardMatches
pkg/engine/ports.go
Ports (Dependency Interfaces)
Defines the three hexagonal ports: PrincipalStore (CRUD + type-filtered list), GrantStore (policy assignment operations), and PrincipalMatcher (pure stateless matching, no DB). Implementations live in separate files (principal_store_gorm.go, principal_matcher.go), allowing easy substitution.
PrincipalStoreGrantStorePrincipalMatcher

Data Access

pkg/store/gorm.go
GORM Principal Store
Implements both PrincipalStore and GrantStore ports (defined in pkg/engine/ports.go) using GORM. Persists Principal records and the principal_policies join table to SQLite (development) or PostgreSQL (production). Handles AutoMigration, index creation, and all context-aware CRUD operations including batch grant/revoke. Satisfies engine interfaces via Go structural typing — no import of pkg/engine needed.
GormPrincipalStoreAutoMigrateListByTypeGrant / Revoke
pkg/store/matcher.go
Principal Matcher
Two stateless PrincipalMatcher implementations. OIDCMatcher filters principals by JWT claim conditions (equals, contains, regex). X509Matcher filters by certificate attributes (serial, CN, CA fingerprint, authority key ID). MatchService coordinates both, loading principals by auth type and dispatching to the appropriate matcher. Imports pkg/engine only for the PrincipalStore and PrincipalMatcher interface types.
OIDCMatcherX509MatcherMatchServiceDefaultMatchService

Data Flows

1 — Authorization Check (Direct Principal)

1
authz_controller.go — receives POST /v1/authz/authorize, binds JSON, validates namespace/schema/entity.
2
pkg/service/identity.go → pkg/store/gorm.go — loads all grants for the given principal ID from SQLite.
3
pkg/service/identity.go → pkg/service/policy.go — fetches each policy JSON blob from the blob store, builds a PolicyRegistry.
4
pkg/engine/engine.go Authorize() — classifies action: global (return immediately based on rule match) or atomic (continue to filter).
5
pkg/engine/filter.go GenerateCheckFilter() → pkg/engine/graph.go — finds cascading access paths, builds JOIN + WHERE conditions.
6
pkg/engine/engine.go — executes SELECT 1 FROM table WHERE … LIMIT 1 against the entity's PostgreSQL DB. Returns allowed=true if rows > 0.
7
pkg/service/authz.go — logs the final decision with trace-ID (from context) at Info level.

2 — Match & Authorize (Auth Material)

1
gin-middleware/middleware.go AuthzCheck() — extracts auth type and credential from Gin context (set by upstream auth middleware).
2
pkg/service/authz.go MatchAndAuthorize() → pkg/service/identity.go Resolve() — dispatches to MatchService.
3
pkg/store/matcher.go MatchService — loads all active principals of the given type from SQLite, feeds to OIDCMatcher or X509Matcher.
4
OIDCMatcher / X509Matcher — pure in-memory filtering: checks JWT claims or certificate attributes against each principal's AuthConfig. Returns matching principal IDs.
5
For each matched principal: load grants → load policy blobs → merge into one PolicyRegistry.
6
engine.go — proceeds as in Flow 1, steps 4–6.

3 — List Filter Generation

1
Caller (middleware or controller) invokes MatchAndGetFilter() or GetFilter().
2
Principal resolution proceeds identically to flows above → PolicyRegistry ready.
3
pkg/engine/engine.go GetListFilter() → pkg/engine/filter.go GenerateListFilter() — for each rule: build IN-clause (direct grants), JOIN chains (cascading relations), or column conditions (attribute filters).
4
Returns a FilterResult with SQL joins and WHERE conditions. Full SQL is logged at Trace level; shape (count) at Debug.
5
Middleware stores the WHERE clause in the Gin context under authz_query. The downstream handler appends it to its GORM query before execution.

4 — Capability Query

1
capabilities_controller.go — receives a batch of entity queries with a principal ID.
2
pkg/service/capabilities.go GetPrincipalPolicies() — loads policies exactly once for the principal via PrincipalManager + PolicyManager.
3
pkg/engine/capabilities.go GetEntityCapabilitiesBatch() — for each query, calls GetEntityCapabilities() which probes each atomic action via a separate Authorize() call.
4
pkg/engine/capabilities.go GetGlobalCapabilities() — scans all schema global actions against all policy rules; no DB query needed.
5
Results returned as a list aligned to the input query order. For auth-material variants, results are OR-merged across all matched principals.
